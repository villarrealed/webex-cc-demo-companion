T3
JDS context append + read — direct from WxCC (all writes + reads)
Read/write spine to CJDS proven end-to-end (Thu 22). Architecture: Option A — all CJDS writes direct from WxCC Flow Designer. Connect orchestration deprecated for this demo.
  • Write path (direct): WxCC HTTP Request node → POST /publish/v1/api/event?workspaceId={wsId} via Webex CC connector. CloudEvents body shape (specversion 1.0, type, source, time, identity, identitytype, data{...}). CJDS returns 202 Accepted. Same connector handles auth/token refresh.
  • Read path: WxCC HTTP Request node (synchronous) → GET /v1/api/events/workspace-id/{wsId}?identity={ANI}&pageSize=1&filter=type=='com.cisco.wxcc.task.routed' via same connector.
  • Payload fields: intent, customerLang, routedQueue, aiOptOut, customerID, vaSummary, vaOutcome, skillsApplied, uiData (division/title/subTitle/iconType/filterTags). Note: booleans serialize as strings ("false") — use string comparison in conditions.
  • Why direct over Connect: single destination (CJDS only), lower latency, fewer hops, easier to debug in WxCC flow logs, demo story is cleaner ("one node writes the summary"). Connect orchestration shines for multi-destination/multi-channel — overkill here.
  • Deprecated: Connect write webhook (events/9FB0GS61XX) + read webhook (events/FVYFSJB20A). Connect inbound webhooks are async-only — no sync response possible. Kept in tenant for reference but not used in demo flow.
RecWire CJDS read as first node after NewPhoneContact. Use $.meta.resultCount for returning-caller branch, $.data[0].data.customerLang for language overlay, $.data[0].data.intent for repeat-reason prompt (T6), $.data[0].data.aiOptOut for AI opt-out check. CJDS writes: one HTTP Request after VA block (resolved + escalated outcomes write same event type com.cisco.wxcc.va.summary with different vaOutcome field). Always add Failure/Timeout branches falling through to default path so flow never breaks on CJDS errors.
Root cause found (Fri 29)Custom event type values do NOT render in the Customer Journey widget by default. Widget renders only platform-native types (task:new, task:connected, task:ended, agent:state_change, wrapup) with built-in renderers. Custom types like com.cisco.wxcc.va.summary are accepted (202), stored in CJDS, and their filterTags appear in the Filter By dropdown — but the entries themselves are silently dropped from the timeline. Also confirmed: iconType: "robot" is NOT in the widget icon map (Momentum Design); use "task". Fix: publish as type: "task:new" with uiData.title, uiData.subTitle, uiData.iconType: "task", uiData.filterTags. Verified working: event 78c69ffa-9bec-4f26-88d5-15b0026e8661 rendered as "AI Virtual Agent Summary" in widget. Sources: CJAAS widgets README v10, icon map.
